06
五、联合工作流

角色分工与执行隔离

多 Agent 不是堆更多模型，而是让不同角色只做自己最该做的那一段工作。

Master Orchestrator
负责任务路由、状态收口和异常升级，不直接写业务代码。
执行与评估分离
Coder 不给自己结论，Reviewer 不承担实现职责
Coder
代码实现 / 跑测试 / 快速修复
负责

实现代码、补测试、响应 Review。

不负责

不能自己宣布“已经没问题”。

Reviewer
代码审查 / 测试审查 / 质量门禁
负责

独立找风险、验证回归、给 RED/YELLOW/GREEN。

不负责

不背实现指标，也不顺手接管编码。

LongRunner
长任务 / 调研 / 中文文档输出
负责

处理长周期任务，提取经验，补充文档。

不负责

不承担高频短链路的即时实现。

为什么要拆角色
避免一个 Agent 同时兼顾协调、实现、审查。
防止自我确认偏差把问题直接带进主线。
让长任务从高频链路里分流出去。
工程化收益
职责清晰
谁推进、谁把关、谁收口一目了然
独立审查
评估结论来自另一个角色而不是作者本人
任务分流
长任务与短链路任务可以并行而不打架
07
六、实践经验

五个踩过的坑，如何长成规则

真正有效的提示词不是先设计好的，而是从真实运行问题里一轮轮跑出来的。

坑 1
Agent 记忆不可靠
同一 issue 跨 run 容易丢上下文，所以补了 PROGRESS.md 和文件化交接。
坑 2
Agent 自己审查自己
Coder / Reviewer 强制分离，执行者不能给自己最终结论。
坑 3
流程卡死没人管
评论 @Master + assign 回 Master，两步汇报缺一不可。
坑 4
迭代失控
代码/测试评审最多 2 轮，超过就自动升级到人。
坑 5
Runtime 挂了 + 429 限速
设定延时重试和次数上限，超过阈值再升级处理。
提示词优化闭环
持续改进 不是“设计”，而是“运行后修订”
运行任务
记录日志
分析失败
修订规则
重跑验证
“提示词不是设计出来的，是跑出来的。”
阿里 / 淘宝 / 美团等经验总结提供基础版本，真实项目运行决定最后的硬规则。
文件化上下文
Issue、jsonl、PROGRESS.md 共同承接跨 run 信息
执行评估分离
质量判断必须来自独立角色而不是作者本人
上限与重试
给迭代次数、重试次数和升级阈值设硬边界
08
七、结论

稳定性来自协作机制，不来自单一模型

把工具、流程和角色拆开，AI 协作开发才会真正进入可复用、可验证、可管理的工程状态。

Multica
多 Agent 协作平台
统一 Issue、角色、状态与本地 Runtime 调度。
Trellis
单 Agent 执行框架
把规范、任务上下文、工作流和日志拆成按需加载的结构。
SDD
结构化任务定义
用 Spec / Plan 固定 WHAT，再把 HOW 交给 Agent 去实现。
一条更稳的工程路径
Spec 约束目标 → Plan 约束实施 → 平台组织协作 → 角色分离控制质量 → 复盘沉淀规范
建议优先落地
先建立 Spec / Plan 习惯
明确 Coder / Reviewer 分离
将关键上下文文件化
再引入平台化调度
